# 数据库优化方案（面向 Codex 执行）

本文档不是抽象原则清单，而是一份可以直接交给 Codex 在本仓库内持续推进的数据库优化路线图。目标不是一次性“大重构”，而是在**不覆盖用户现有修改**、**不破坏前端响应结构**、**优先保护在线主链**的前提下，分阶段把数据库层从“可用但混杂”推进到“结构稳定、行为可预期、便于后续维护”。

适用仓库根目录：

- `D:\Desktop\Intelligent-Edu-Platform`

核心约束：

- 只用本地代码和命令行，不用浏览器插件。
- 如仓库根目录存在 `.codegraph/`，优先用 CodeGraph 理解代码与调用链。
- 不回滚或覆盖用户已有修改。
- 不为了“风格统一”去做大面积无关改动。
- 所有改动以主链安全为先，优先处理在线路径，脚本和一次性迁移保守推进。
- 任何会影响前端 JSON shape 的改动都必须视为高风险。

---

## 1. 当前数据库现状判断

### 1.1 总体评价

当前数据库层已经具备比较明确的工程化基础：

- 有统一的 Mongo client 创建逻辑，且已启用 `tz_aware=True`
- 有集中式索引入口：`backend/core/database.py`
- 新模型已经从 legacy 嵌套结构迁移出一批平铺集合：
  - `course_sections`
  - `enrollments`
  - `assignments`
  - `submissions`
  - `documents`
  - `grades`
- chat 侧已经建立了专门集合和容量控制：
  - `chat_rooms`
  - `chat_messages`
  - `chat_file_transfers`
  - `ai_chat_sessions`
  - `ai_session_buckets`
- 已经开始引入共享 helper：
  - `backend/repositories/_helpers.py`

这说明数据库并不是“失控状态”，而是处于**新旧模型并存、主链逐步收敛、风格尚未完全统一**的阶段。

### 1.2 当前主要问题

当前最核心的问题不是“数据库不能跑”，而是**一致性成本偏高**：

1. 新旧模型并存  
   仍然存在 legacy `courses`、legacy homework、部分旧风格历史数据读写逻辑。

2. ObjectId 处理风格不统一  
   仓库里同时存在：
   - shared helper 风格：`coerce_object_id` / `require_object_id`
   - 直接 `ObjectId(...)`
   - `ObjectId.is_valid(...)`
   - 各处零散 `safe_object_id(...)`

3. 时间字段风格混杂  
   当前既有 camelCase：
   - `createdAt`
   - `updatedAt`
   - `submittedAt`

   也有 snake_case：
   - `created_at`
   - `updated_at`
   - `expires_at`

   这并不一定要强行统一成一种命名，但需要统一**时间语义**：
   - 统一 aware UTC
   - 统一比较规则
   - 统一 Mongo 读回后的兼容处理

4. 服务层仍有不少直接触库  
   数据访问虽然已经开始 repo 化，但仍未彻底形成边界，导致：
   - 相同规则在多处散落
   - ObjectId/时间处理重复
   - schema 迁移更难做

5. 少量查询仍带固定上限读取习惯  
   典型表现是：
   - `to_list(500)`
   - `to_list(length=5000)`

   这类写法适合明确分页，不适合“读取全集”的逻辑。

### 1.3 已知已经推进过的方向

截至当前，数据库收敛工作已经在以下方向上取得进展：

- 一批在线主路径的固定上限读取已被改成游标全量读取
- 一批主链服务已经切到 `backend/repositories/_helpers.py`
- `ai_session_buckets`、chat transfer、file assets、AI session 等链路开始统一 aware UTC / ObjectId helper
- 已经补了一部分回归测试，尤其是：
  - AI session
  - file assets lifecycle
  - database indexes
  - chat database convergence

这意味着后续工作不应推倒重来，而应按“剩余尾项收口 + 旧链路隔离 + repo 边界继续推进”的方式进行。

## 2. 优化总目标

最终目标不是“把所有数据库代码都改成一种样子”，而是实现以下几件事：

1. 在线主链不再依赖零散的 `ObjectId(...)` 与 naive UTC 写法
2. 读取策略明确：分页就分页，全集就显式游标
3. 关键集合的 schema 语义稳定，行为被测试保护
4. legacy 路径明确标注、隔离、收缩，不再与主链混写
5. 后续 Codex 接手时，不需要每次重新发明数据库约束

---

## 3. 执行原则

所有 phase 都必须遵循以下原则。

### 3.1 优先级原则

按以下顺序推进：

1. 在线主链写路径
2. 在线主链读路径
3. TTL / 时间逻辑
4. repository/helper 收敛
5. legacy 脚本 / backfill / 一次性迁移
6. 风格层统一

### 3.2 修改原则

- 优先复用已有 helper，不新造第二套 helper。
- 优先小步改动，不做跨十几个文件的大扫除。
- 一个 phase 只解决一类问题，不混入无关整理。
- 不要改变前端依赖的字段名和响应结构，除非该 phase 明确允许。

### 3.3 测试原则

- 补测试时优先测行为，不优先测实现细节。
- 任何涉及时间逻辑的改动，尽量断言 aware UTC 或兼容旧文档。
- 任何涉及 membership / 权限 / ownership 的改动，至少测一条负例。
- 优先用现有测试风格，不新增笨重测试框架。

### 3.4 Codex 执行原则

Codex 在执行每个 phase 前，应先做：

1. 用 CodeGraph 理解目标文件和调用链
2. 看 `git status --short`
3. 识别该 phase 是否触及在线主链
4. 找现有测试覆盖点

Codex 在执行每个 phase 后，应输出：

1. 具体改了哪些文件
2. 跑了哪些 pytest
3. 还剩哪些尾项
4. 哪些点故意没动，为什么

## 4. Phase 0：建立数据库收敛基线

### 目标

在不改业务逻辑的前提下，先把“数据库问题清单”结构化，避免后续每次重复摸底。

### 范围

- `backend/core/database.py`
- `backend/repositories/_helpers.py`
- `backend/repositories/`
- `backend/services/`
- `backend/routes/`

### 要做的事

1. 统计并归类以下模式：
   - 直接 `ObjectId(...)`
   - `ObjectId.is_valid(...)`
   - `datetime.utcnow()`
   - `datetime.now(timezone.utc)`
   - `to_list(500)` / `to_list(length=5000)` 等固定上限
   - 直接 `from backend.core.database import db`

2. 把结果分成三类：
   - 在线主路径
   - 边缘管理路径
   - legacy/脚本/backfill

3. 建立执行白名单，避免后面范围失控。

### 完成定义

- 能明确说出“先动哪些，不动哪些”
- 后续每个 phase 都能在这个基线上继续推进

## 5. Phase 1：ObjectId 收敛（在线主链优先）

### 目标

把在线主链里零散的 ObjectId 转换逻辑统一到共享 helper，降低运行时行为漂移。

### 优先使用的 helper

文件：

- `backend/repositories/_helpers.py`

优先复用：

- `coerce_object_id(value)`
- `require_object_id(value, detail=...)`
- `oid(doc)`

### 适用规则

1. **读路径**
   - 如果非法 id 应该返回空结果：优先 `coerce_object_id`
   - 如果非法 id 应该直接报错：优先 `require_object_id`

2. **写路径**
   - 用户输入直接参与更新/删除时，优先 `require_object_id`

3. **HTTP 入口**
   - 如果历史行为是 `400`，则 helper 失败时必须保留 `400`

### 优先文件

高优先：

- `backend/services/chat_service/query_service.py`
- `backend/services/chat_service/room_service.py`
- `backend/services/chat_service/message_service.py`
- `backend/services/chat_service/transfer_dispatch_service.py`
- `backend/services/chat_service/session_bucket_service.py`
- `backend/services/files/file_center_service.py`
- `backend/services/file_assets/queries.py`
- `backend/services/file_assets/lifecycle.py`
- `backend/services/ai/ai_session_service.py`

中优先：

- `backend/services/student/student_assignment_service_support/*.py`
- `backend/repositories/history_repo.py`
- `backend/repositories/user_repo.py`
- `backend/repositories/homework_repo.py`

低优先：

- `backend/scripts/*.py`
- `backend/services/file_assets/backfill_ai_chat.py`

### 不要做的事

- 不要顺手把所有字符串外键都改成 ObjectId 存储
- 不要因为 helper 化而改变已有响应结构
- 不要在一个 phase 里同时重构 repo 边界和业务逻辑

### 建议测试

- 非法 id 返回 `400` 的接口/服务测试
- 合法 id 查询仍命中 ObjectId 文档
- 非法 id 的“空结果”路径不抛异常

### 完成定义

- 在线主链基本不再直接依赖 `ObjectId(...)` / `ObjectId.is_valid(...)`
- 非法 id 行为在测试中被锁住

## 6. Phase 2：时间语义收敛（aware UTC 优先）

### 目标

统一数据库时间写入与比较语义，优先解决 naive/aware 混用带来的运行期隐患。

### 优先使用的 helper

- `backend/repositories/_helpers.py: utcnow()`
- 某些局部域 helper，如：
  - `backend/services/file_assets/shared.py: utcnow()`

### 原则

1. 所有新写入时间统一使用 aware UTC
2. 读取旧文档时允许兼容 naive datetime
3. TTL 字段优先保留字段名，不强行统一命名
4. 时间比较逻辑显式处理 legacy naive 文档

### 高风险场景

- transfer ticket 过期判断
- auth/session TTL
- background job lease
- AI session / chat message 时间线
- 任何序列化到前端的时间字段

### 优先文件

- `backend/services/chat_service/transfer_dispatch_service.py`
- `backend/services/chat_service/query_service.py`
- `backend/services/ai/ai_session_service.py`
- `backend/services/auth/*.py`
- `backend/repositories/*_repo.py`
- `backend/core/database.py`

### 典型处理方式

1. 写入：
   - `datetime.utcnow()` -> `utcnow()`

2. 比较：
   - 对旧文档读取出来的 naive `datetime`，先显式归一到 UTC 再比较

3. 序列化：
   - 保持前端现有字段名
   - 只统一其底层时区语义

### 建议测试

- 新写入的 datetime `tzinfo == timezone.utc`
- legacy naive `expires_at` 仍然能正确过期
- 序列化结果保持与前端兼容

### 完成定义

- 主链时间写入全部为 aware UTC
- 主链比较逻辑不再依赖“Mongo 会帮我凑合”

## 7. Phase 3：读取策略收敛（分页与全集读取分离）

### 目标

消除“看起来像全集读取，其实被固定上限截断”的风险。

### 问题模式

以下写法需要逐步清理：

- `to_list(500)`
- `to_list(length=500)`
- `to_list(length=5000)`

### 原则

1. 明确分页场景：保留 `limit/page/page_size`
2. 明确全集读取场景：用游标全量遍历
3. 不要把“历史上先写个 500 防炸”当成正式逻辑

### 适合保留固定 limit 的场景

- 管理后台人工列表页
- 明确分页接口
- 受前端参数约束的查询

### 必须改为游标全集的场景

- 迁移脚本
- 聚合统计的完整数据读取
- 后台同步/回填任务
- 全量扫描型后台处理

### 优先文件

- `backend/scripts/migrate_to_v2.py`
- `backend/infrastructure/telemetry.py`
- `backend/repositories/history_repo.py`
- `backend/services/rag_service/*`
- `backend/services/grading_service/*`

### 建议 helper

统一维护少量局部 helper，例如：

- `_read_all(cursor)`

不要每个文件发明不同版本的“读全集”函数。

### 建议测试

- 构造超过历史上限数量的数据，确认不再截断
- 对分页接口确认 page_size 仍然生效

### 完成定义

- 代码里不再把固定上限伪装成“全量读取”

## 8. Phase 4：repository 边界继续推进

### 目标

减少 service 层直接触库，让数据库规则集中在 repo/helper 中表达。

### 为什么要做

当前很多 service 直接 `from backend.core.database import db`，短期方便，长期代价是：

- ObjectId 规则分散
- 时间写法分散
- schema 变更代价高
- 测试替身更难做

### 不是要做什么

不是把所有 service 一夜之间全部 repo 化。

### 要做什么

选主链、分批推进：

第一批适合推进的域：

- AI session
- file assets
- chat rooms / messages 查询
- submissions / documents / enrollments / assignments

### repo 化的最小标准

1. 把重复出现的查询条件沉到 repo
2. 把 ObjectId / 时间处理沉到 repo/helper
3. service 只保留业务编排，不保留数据库规则拼装

### 优先候选文件

- `backend/services/chat_service/query_service.py`
- `backend/services/chat_service/room_service.py`
- `backend/services/files/file_center_service.py`
- `backend/services/file_assets/*.py`
- `backend/services/student/student_assignment_service_support/*.py`

### 不要做的事

- 不要为了 repo 化拆出大量只有一行的薄封装
- 不要在同一次提交里既重命名又搬迁又改逻辑

### 完成定义

- 主链 service 的数据库规则显著减少
- repo/helper 成为数据库约束的主要落点

## 9. Phase 5：legacy 路径隔离与收缩

### 目标

明确 legacy 路径存在的理由，避免其继续污染主链。

### legacy 的典型来源

- `courses`
- legacy homework / `homework_submissions`
- 一次性迁移脚本
- backfill 脚本
- 兼容旧前端或旧数据格式的 fallback

### 策略

1. **在线主链需要调用的 legacy**
   - 可以保留
   - 但要单独集中在 support/legacy 模块中

2. **只在脚本里存在的 legacy**
   - 保守处理
   - 优先补注释和边界说明

3. **已不再需要的 legacy fallback**
   - 先确认无调用与无数据依赖
   - 再清理

### 建议做法

- 在文件名和函数名上显式标明 `legacy`
- 在关键 fallback 入口补一句注释，说明为何仍然保留
- 对 legacy 与 v2 双读路径补行为测试

### 优先关注文件

- `backend/services/student/student_assignment_service_support/legacy_homework.py`
- `backend/scripts/migrate_to_v2.py`
- `backend/services/file_assets/backfill_ai_chat.py`
- `backend/services/homework/homework_service.py`

### 完成定义

- Codex 再读代码时，能明显区分“主链”与“兼容旧链路”

## 10. Phase 6：索引与查询契约复审

### 目标

确认代码里实际使用的查询模式，和 `ensure_indexes()` 里的索引设计一致。

### 重点集合

- `users`
- `user_sessions`
- `course_sections`
- `enrollments`
- `assignments`
- `submissions`
- `documents`
- `chat_rooms`
- `chat_messages`
- `ai_chat_sessions`
- `chat_file_transfers`
- `file_assets`
- `background_jobs`
- `indexing_jobs`

### 复审重点

1. 代码真实查询字段是否被索引覆盖
2. 是否有只排序不筛选的热点查询
3. 唯一索引是否与业务契约一致
4. TTL 索引是否和代码中的时间字段保持一致
5. legacy 索引是否还能减掉

### 典型复审案例

1. `chat_rooms`
   - `members`
   - `courseId + type`
   - `directPairKey`

2. `chat_messages`
   - `roomId + sentAt`
   - `roomId + readBy + senderId`

3. `file_assets`
   - `file_type + status + created_at`
   - `owner_type + owner_id`
   - `session_id + created_at`
   - `conversation_date + user_id + created_at`

4. `chat_file_transfers`
   - `transfer_id`
   - `owner_user_id + created_at`
   - `status + expires_at`

### 测试

- 继续维护并扩充：
  - `backend/tests/test_database_indexes.py`

### 完成定义

- 关键查询都能对应到清晰索引
- 索引文件成为数据库契约文档的一部分

## 11. Phase 7：测试护栏补齐

### 目标

把数据库收敛成果从“代码风格”变成“测试护栏”。

### 优先补哪些测试

1. ObjectId 行为测试
   - 合法 id 命中
   - 非法 id 返回空/报错符合旧契约

   - 新写入时间带 `timezone.utc`

3. 分页/全集读取测试
   - 超过历史上限仍不截断

4. chat 主链行为测试
   - membership 校验
   - direct room / course group room 创建
   - transfer 过期
   - bucket 追加与回放

5. file assets 行为测试
   - `_id` / `file_id` 回退
   - 软删/恢复/硬删引用检查

### 已有可复用测试文件

- `backend/tests/test_ai_session_service.py`
- `backend/tests/test_file_assets_lifecycle.py`
- `backend/tests/test_database_indexes.py`
- `backend/tests/test_backend_refactor_roundup.py`
- `backend/tests/test_chat_database_convergence.py`

### 完成定义

- 每个 phase 至少有一组直接覆盖的 pytest
- 收敛后续不再轻易回退

## 12. 建议执行顺序

推荐按下面的顺序推进，而不是乱序穿插：

1. Phase 0：建立基线
2. Phase 1：ObjectId 收敛
3. Phase 2：时间语义收敛
4. Phase 3：读取策略收敛
5. Phase 7：补测试护栏
6. Phase 4：repo 边界继续推进
7. Phase 5：legacy 隔离与收缩
8. Phase 6：索引与查询契约复审

原因：

- 先稳主链运行时风险
- 再稳行为测试
- 最后才做边界与结构层优化

建议 Codex 每次按下面模板汇报：

- 哪些刻意没动
- 下一轮建议先做什么

## 14. 明确不建议做的事

以下事项不建议作为数据库优化 phase 的默认动作：

1. 一次性重命名所有 `created_at` / `createdAt`
2. 一次性把所有 service 全部改成 repository
3. 把所有字符串外键强推成 ObjectId
4. 为了“统一风格”去改前端响应字段
5. 在没有测试护栏前做跨域大重构
6. 因为脚本文件看起来旧，就直接删除 legacy fallback

这些事情不是永远不能做，而是不适合作为当前阶段的默认推进方式。

## 17. 给 Codex 的最后说明

如果 Codex 接手执行本文档，请始终遵守下面这组判断：

- 这是一个“逐步收敛”任务，不是“一次性推平”任务。
- 先保护行为，再优化结构。
- 先处理主链，再处理边缘。
- 先用现有 helper，再考虑抽象。
- 任何看起来像“顺手一起改了吧”的冲动，都应先压住。

真正好的数据库优化，不是改得多，而是：

- 风险小
- 可验证
- 可回顾
- 后续的人接手时，不需要重新理解一遍历史遗留。
